## 一、执行摘要

### 总结结论

这是一个已经形成完整闭环的本地英语学习桌面应用，而不是演示型项目。当前仓库已经具备：

- 文章阅读与标注
- EPUB 书架与原生阅读器
- 非阻塞 AI 查词 / 查句
- 句法结构解析
- 生词本 / 句库 / 锚点回跳
- FSRS 间隔复习
- OCR 导入
- 数据导入导出
- TTS 本地缓存

从产品完成度看，项目已进入 **可持续迭代阶段**；从工程风险看，当前主要问题不在功能缺口，而在：

1. **HTML 渲染安全边界过宽**
2. **敏感配置明文落库且被全量导出**
3. **数据备份模型不完整**
4. **Reader 复杂度快速上升，但缺少自动化测试护栏**

### 评级概览

| 维度 | 评级 | 结论 |
|---|---|---|
| 产品完整度 | **B+** | 核心学习闭环已形成，功能组合具有辨识度 |
| 架构分层 | **B+** | Vue composables / Tauri commands / repositories 分层清楚 |
| 工程稳健性 | **B-** | 当前可构建，但缺测试、Reader 复杂度偏高 |
| 安全边界 | **C** | `v-html`、明文配置、宽文件权限三者叠加构成显著风险 |
| 前端性能余量 | **C** | 大 chunk、全局 `KeepAlive`、重 Reader 页面并存 |
| 数据完整性 | **C** | “导出全部数据”与真实数据模型不一致 |

### 置信度标注

- **高置信度**：版本、工作树状态、构建结果、权限配置、导入导出字段、`v-html` 分布
- **中置信度**：Reader 长期维护成本、潜在内存驻留风险、运行时卡顿趋势
- **低到中置信度**：在恶意输入下的完整攻击链可达性；当前报告未做 exploit 复现

## 三、优势与成熟度判断

### 1. 这个项目已经不是“功能散点集合”

当前仓库最强的地方不是某一个单点功能，而是这些能力已经串成闭环：

`阅读 -> 查询 -> 保存 -> 回跳 -> 复习 -> 再阅读`

这使它明显区别于：

- 只做翻译的工具
- 只做标注的工具
- 只做 Anki 卡片的工具

### 2. 分层方式适合当前阶段

当前分层没有明显失控：

- 前端把路由页、组件、composables、services 分开
- 后端把 command 与 repository 分开
- SQLite schema 迁移也已有版本管理

对于单机本地桌面应用，这套结构是够用的。  
问题不在“架构错误”，而在“功能继续堆叠后，复杂视图正在变成维护热点”。

### 3. 产品方向是对的

从英语学习者视角看，当前最有价值的组合是：

- **非阻塞 quick lookup**
- **句法结构解析**
- **EPUB 图书阅读器**
- **FSRS 复习**

这条路线明显比“单纯翻译器”更有长期产品价值。

## 四、Critical / High 问题

以下部分按“事实 -> 风险 -> 建议”组织。

### 🔴 S1. 多处 `v-html` 直接挂载，且未见统一 sanitize

#### 事实

当前 live repo 仍有多处直接 `v-html`：

- `src/components/QuickLookupPanel.vue`
- `src/views/ReviewView.vue`
- `src/views/SentencesView.vue`
- `src/components/AnnotationForm.vue`
- `src/components/ArticleWordList.vue`
- `src/components/ArticleReader.vue`
- `src/views/EpubImportView.vue`
- `src/components/ArticleListPanel.vue`
- `src/views/GuideView.vue`

同时，代码中未检出：

- `DOMPurify`
- 统一 `sanitize*` util
- 明显的 HTML 白名单过滤层

#### 风险判断

这不是“所有 `v-html` 都同等级危险”，但至少存在两类高风险输入：

1. **AI 返回的 `parsed_html`**
2. **用户导入的 Markdown / EPUB / OCR 内容转 HTML 后再渲染**

在这两类路径上，如果没有过滤：

- LLM 被 prompt injection 诱导生成恶意标签 / 属性
- 导入内容本身携带危险 HTML

都可能把不可信内容送进 WebView DOM。

#### 建议

建立两层策略：

1. **结构化解析结果白名单过滤**
   - 仅允许 `span`
   - 仅允许句法着色需要的 `class`

2. **通用富文本渲染过滤**
   - Markdown/EPUB/OCR 转出的 HTML 统一进入 sanitize 层

#### 结论

这项问题应列为 **当前第一优先级之一**。

---

### 🔴 S2. `api_key` 明文存储在 `settings`，且被数据导出直接包含

#### 事实

- `src-tauri/src/commands/settings.rs`：设置项直接写入 `settings(key, value)`
- `src/views/SettingsView.vue`：直接读写 `api_key`
- `src-tauri/src/commands/data.rs`：`export_all_data` 直接导出整个 `settings` 表

#### 风险判断

当前行为等价于：

1. API Key 明文落库到 `~/.shiyu/shiyu.db`
2. 用户使用“导出全部数据”时，敏感配置会一起导出

这意味着风险不仅是“本机泄露”，还包括：

- 用户把备份文件发给别人
- 误上传到云盘 / Git
- 调试时把导出文件当普通备份共享

#### 建议

最低限度：

- `export_all_data` 改为 **settings 白名单导出**
- 明确排除：`api_key`、可能的 OCR token、未来所有 secret

更稳方案：

- API Key 迁移到系统凭据存储或 Stronghold
- 数据库里只保留“是否已配置”或非敏感配置

#### 结论

这项问题应与 S1 并列 **P0**。

---

### 🔴 S3. 当前“导出全部数据”并不完整，且与真实数据模型不一致

#### 事实

当前 `ExportData` 只包含：

- `vocabulary`
- `sentences`
- `settings`
- `articles`

但当前数据库与业务模型还包含：

- `ebooks` 表
- `articles.mindmap_markdown`

已知证据：

- `src-tauri/src/db.rs` 有 `mindmap_markdown`
- `src-tauri/src/commands/articles.rs` 有 `get_article_mindmap` / `save_article_mindmap`
- `src-tauri/src/commands/ebooks.rs` / `get_ebooks` 已在实际使用
- `src-tauri/src/commands/data.rs` 既不导出 `ebooks`，也不导出 `mindmap_markdown`

#### 风险判断

当前“导出全部数据”与实际含义不一致，会造成：

- 恢复后书架缺失
- 生词 / 句子的 ebook 锚点无法完整回跳
- 思维导图数据丢失
- `replace` 导入不能形成真正对等恢复

这是**数据完整性问题**，不是单纯体验问题。

#### 建议

`ExportData` 至少应扩展到：

- `ebooks`
- `article_mindmaps` 或在 `ArticleItem` 中包含 `mindmap_markdown`

同时明确：

- `replace` 模式对 `ebooks` 的覆盖策略
- 若不打算导出实体 EPUB 文件，也要在 UI 文案中写清楚“仅导出元数据，不导出图书文件本体”

#### 结论

这项问题优先级应至少到 **P0 / P1 边界**，高于多数 Warning 项。

### 🟠 H1. 文件系统权限面过宽，且图书阅读器允许脚本内容

#### 事实

当前配置包含：

- `src-tauri/capabilities/default.json`
  - `fs:allow-read-text-file` -> `**`
  - `fs:allow-write-text-file` -> `**`
  - `fs:allow-read-file` -> `**`
  - `fs:allow-write-file` -> `**`
  - `fs:allow-exists` -> `**`
  - `fs:allow-mkdir` -> `**`

- `src-tauri/tauri.conf.json`
  - `assetProtocol.scope` -> `["**"]`
  - `bundle.targets` -> `"all"`

- `src/components/BookReader.vue`
  - `allowScriptedContent: true`

#### 风险判断

单独看其中任一项，不一定构成事故；但三者组合后：

- 不可信内容渲染面较大
- 本地文件访问面较大
- EPUB 内部内容执行边界较宽

这会放大 S1 的后果。

#### 需要纠正的表述

本报告只确认了：

- 文件权限面过宽
- `allowScriptedContent` 已开启

**未确认**：

- 当前已具备任意 shell 执行链
- 当前一定可被利用到完整本地接管

因此更准确的说法是：  
这是一个**需要尽快收窄的高风险结构面**，而不是已证明可达的完整 exploit。

#### 建议

- 把文件权限缩到最小业务范围
- 重新评估 `allowScriptedContent: true` 是否必需
- 明确区分“用户导入文件可读”和“应用全盘可读写”的权限边界

### 🟠 H2. AI / OCR 请求未见统一 timeout 与 cancel

#### 事实

当前仍检出多处：

- `reqwest::Client::new()`

主要位置：

- `src-tauri/src/commands/ai.rs`
- `src-tauri/src/commands/ocr.rs`

代码中未见统一：

- `timeout(...)`
- `connect_timeout(...)`
- 用户侧取消机制

#### 风险判断

当上游 API 卡死、代理异常、流式接口长时间无响应时：

- 前端 loading 可能长期悬挂
- 用户只能重试或重启
- 错误恢复路径弱

#### 建议

- 封装统一 HTTP client builder
- 加 connect timeout / request timeout
- 流式接口增加 cancel 事件

### W3. 全局 `KeepAlive` 对重型页面不够克制

#### 事实
